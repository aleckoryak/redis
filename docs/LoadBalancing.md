Когда мы говорим о **балансировке нагрузки (Load Balancing) между Java-приложением и Redis в Production**, классический
подход с использованием HTTP-балансировщиков (вроде Nginx, HAProxy или AWS ALB) **не работает**.

Redis работает по собственному бинарному протоколу RESP (REdis Serialization Protocol), а соединения между Java (через
Lettuce/Redisson) и Redis являются постоянными (длинные TCP-соки / Persistent Connections).

В промышленной разработке балансировка и высокая доступность (HA) реализуются на одном из трех уровней: **на стороне
клиента (Java)**, **через прокси** или **на уровне топологии самого Redis**.

Давай детально разберем 3 основных production-сценария.

---

## Вариант 1. Redis Cluster (Архитектурный стандарт)

В этой топологии у тебя нет одного сервера Redis. Нагрузка шардируется (распределяется) между несколькими
мастер-узлами (минимум 3 мастера + 3 реплики). Данные делятся на 16384 хэш-слота.

### Как работает балансировка:

* **Умный клиент:** Балансировщиком выступает сама Java-библиотека (**Lettuce** или **Redisson**). При старте приложения
  клиент отправляет команду `CLUSTER SLOTS`, скачивает карту распределения данных и «знает», какой мастер-узел за какой
  диапазон ключей отвечает.
* **Прямая маршрутизация:** Если Java-приложению нужен ключ `user:123`, Lettuce вычисляет CRC16 от ключа, понимает, что
  этот слот лежит на `Master-2`, и отправляет TCP-запрос **напрямую** на `Master-2`, минуя любые промежуточные прокси.
* **Балансировка чтения (Read-From-Replica):** В Lettuce можно настроить свойство `ReadFrom.REPLICA_PREFERRED`. Тогда
  все команды записи (`SET`, `DEL`) клиент будет слать на Мастера, а тяжелые команды чтения (`GET`, `MGET`) равномерно
  распределять по репликам, разгружая основные узлы.

**Как настроить в Java (Spring Boot):**
Тебе достаточно указать список *всех или нескольких* узлов кластера в `application.yml`. Lettuce сам соберет топологию:

```yaml
spring:
  data:
    redis:
      cluster:
        nodes:
          - 10.0.0.1:6379
          - 10.0.0.2:6379
          - 10.0.0.3:6379

```

## Вариант 2. Проксирование через Envoy или HAProxy (TCP/RESP уровень)

Если по каким-то причинам нельзя использовать Redis Cluster (например, вы используете обычную репликацию Master-Slave)
или вы хотите полностью скрыть топологию базы данных от Java-приложения, перед Redis ставится специализированный
прокси-сервер.

В современном Enterprise для этого чаще всего используют **Envoy Proxy** или **HAProxy** (в режиме `mode tcp`).

### Как работает балансировка:

1. Java-приложение ничего не знает про реальные IP-адреса Redis. Оно подключается к одному эндпоинту — прокси-серверу (
   например, `redis-proxy.internal:6379`).
2. Прокси-сервер держит пулы соединений к реальным серверам Redis.
3. **Envoy** обладает встроенным фильтром `Envoy Redis Filter`. Он умеет парсить протокол RESP, отслеживать здоровье
   узлов и распределять запросы (например, Round-Robin). Если один из Redis-серверов падает, Envoy моментально исключает
   его из балансировки, а Java-приложение этого даже не замечает (для Java соединение с прокси остается стабильным).

**Плюс:** Идеально для service mesh архитектур. Клиентский код в Java остается максимально простым.
**Минус:** Появляется дополнительный сетевой хоп (Network Hop), что добавляет около 0.5–1 мс к задержке (latency). Для
In-memory базы это иногда критично.

## Вариант 3. Redis Sentinel (Высокая доступность + Балансировка чтения)

Паттерн применяется, когда данных немного (они помещаются на один сервер), но нужна высокая доступность. У тебя есть
один **Master** (для записи и чтения) и несколько **Slaves/Replicas** (только для чтения). За их здоровьем следят
процессы **Sentinel**.

### Как работает балансировка:

* **Запись:** Все операции модификации данных Java отправляет строго на текущий Master.
* **Чтение (Балансировка):** На уровне конфигурации Lettuce/Redisson включается балансировка чтения по репликам. Клиент
  периодически опрашивает Sentinel-узлы (`Получить адрес текущего мастера/слейвов`), получает актуальный пул реплик и
  балансирует запросы `GET` между ними.

## Производственные рекомендации (Production Checklist)

1. **Если у вас AWS/Cloud:** Используйте **AWS ElastiCache (Redis OSS)** в режиме Cluster Mode Enabled. AWS сам берет на
   себя управление узлами, а в Java вы настраиваете Lettuce на один Configuration Endpoint. AWS балансирует нагрузку на
   сетевом уровне + Lettuce маршрутизирует слоты.
2. **Настройка пулинга в Lettuce:** По умолчанию Lettuce использует одно соединение. Для высоконагруженных систем (сотни
   тысяч запросов) этого мало из-за ограничений пропускной способности одного TCP-канала. Обязательно настраивайте пул
   соединений:

```java
// Пример тюнинга Lettuce pool для высокой утилизации
GenericObjectPoolConfig poolConfig = new GenericObjectPoolConfig();
poolConfig.

setMaxTotal(64);
poolConfig.

setMaxIdle(32);
poolConfig.

setMinIdle(8);

```

3. **Обработка команд `MULTI/EXEC` (Транзакции):** Если вы используете Redis Cluster, помните, что транзакции и
   множественные операции (`MGET`/`MSET`) работают **только в пределах одного хэш-слота**. Если вы попытаетесь сделать
   `MGET` по ключам, которые улетели на разные мастера, Java выбросит
   `RedisCommandExecutionException: CROSSSLOT Keys in request don't hash to the same slot`.

* *Решение:* Использовать хэш-теги в именах ключей, например, `user:{123}:profile` и `user:{123}:settings`. Всё, что
  внутри `{}`, гарантированно попадет на один и тот же сервер Redis, и балансировка не сломает логику.

Использование внешних прокси-серверов (**Envoy Proxy** или **HAProxy**) совместно с **Redis Cluster** — это особая и
довольно сложная архитектурная тема.

Здесь кроется главный подвох: **Redis Cluster по своей природе спроектирован так, чтобы работать БЕЗ прокси.** Он
ожидает, что смарт-клиент (например, Java Lettuce) напрямую общается со всеми узлами кластера.

Тем не менее, в Enterprise-архитектуре (особенно в Cloud/Kubernetes и Service Mesh окружениях) прокси перед Redis
Cluster все равно ставят. Давай детально разберем, как с этим справляются Envoy и HAProxy, и какой вариант выбрать.

---

## 1. Envoy Proxy + Redis Cluster (Рекомендуемый выбор)

Envoy — это cloud-native прокси, который обладает **глубоким пониманием протокола RESP (Redis)** и имеет встроенный
специальный фильтр: `envoy.filters.network.redis_proxy`.

### Как это работает:

Envoy умеет работать в режиме **Cluster Discovery**. При старте он сам подключается к Redis Cluster, выполняет команду
`CLUSTER SHARDS` (или `CLUSTER SLOTS`), строит у себя в памяти карту хэш-слотов и постоянно следит за её актуальностью.

1. Твое Java-приложение думает, что общается с *одним* обычным сервером Redis (подключается к Envoy по адресу
   `localhost:6379` или через Kubernetes Service).
2. Приложение отправляет команду, например: `SET user:99 "data"`.
3. Envoy на лету парсит эту команду, вычисляет хэш-слот для ключа `user:99`, определяет, на каком именно мастере в Redis
   Cluster лежит этот слот, и **сам перенаправляет запрос на нужный узел**.
4. Для Java-приложения весь кластер скрыт за одним эндпоинтом.

### Плюсы Envoy:

* **Абсолютная прозрачность для разработчиков:** В Java коде тебе не нужно настраивать Lettuce в режим кластера. Ты
  настраиваешь его как "Standalone" (одиночный сервер), смотрящий на Envoy. Всю сложность шардинга Envoy берет на себя.
* **Управление CROSSSLOT операциями:** Envoy умеет агрегировать некоторые мульти-ключевые операции, если они
  распределены по кластеру.
* **Метрики:** Из коробки выдает тонну метрик в Prometheus (latency команд, количество ошибок, аптайм узлов).

## 2. HAProxy + Redis Cluster (Вариант с ограничениями)

HAProxy — это невероятно быстрый и стабильный балансировщик, но он работает преимущественно на **L4 (TCP уровень)**. Он
не умеет «заглядывать» внутрь команд Redis и динамически перенаправлять запросы на основе хэш-слотов ключа.

### Как это работает (и почему это больно):

Если ты просто поставишь HAProxy в режиме `mode tcp` перед Redis Cluster и настроишь round-robin балансировку по всем
мастерам:

1. Java-приложение шлет запрос `SET user:99 "data"` в HAProxy.
2. HAProxy перенаправляет этот TCP-пакет на случайный узел, например, `Master-1`.
3. Если этот хэш-слот реально принадлежит `Master-1` — всё супер, команда выполнится.
4. Но если этот слот лежит на `Master-3`, сам Redis вернуть ошибку-редирект: `-MOVED 12345 10.0.0.3:6379`.
5. Если твой Java-клиент настроен как Standalone, он не поймет этот редирект и выбросит ошибку в лог. Если клиент
   настроен как Cluster, он попытается пойти напрямую на `10.0.0.3:6379` в обход HAProxy (что сводит на нет смысл
   прокси).

### Как заставить HAProxy работать с Redis Cluster правильно?

HAProxy используют перед Redis Cluster только в двух специфических кейсах:

1. **Множество Frontend-эндпоинтов:** В конфигурации HAProxy создается $N$ фронтендов (портов) — по одному на каждый
   узел Redis Cluster (например, порт 7001 смотрит на Master-1, порт 7002 на Master-2...). Смарт-клиент (Lettuce)
   подключается к HAProxy, указывая все эти порты. HAProxy здесь выступает просто как статическая точка входа (
   IP-виртуализация) и защитник от падения конкретного узла (health check переключает трафик на реплику).
2. **Только чтение (Read-Only Load Balancing):** HAProxy балансирует запросы только на чтение (`GET`) по репликам
   кластера, используя встроенные проверки `tcp-check` для отправки команды `READONLY` при установке соединения.

## Архитектурный вердикт

* Если перед вами стоит задача **интегрировать Redis Cluster в Kubernetes / Service Mesh** или вы хотите полностью
  избавить Java-разработчиков от поддержки конфигурации кластера в коде — **однозначно выбирайте Envoy**. Его
  Redis-фильтр создавался именно под эту задачу.
* Если у вас **классическая on-premise инфраструктура (виртуалки/железо)**, данные не влезают в один сервер и вы
  используете современный Java-клиент (Lettuce/Redisson), то **лучший балансировщик — это сам Java-клиент**. Уберите
  прокси вообще. Дайте Lettuce прямой доступ к узлам Redis Cluster. Это обеспечит минимальный network latency ($<0.5$
  мс) и максимальную надежность без лишних промежуточных звеньев.
