## 1. Strings (Строки)

Самый базовый и часто используемый тип. Может хранить текст, сериализованный объект (JSON/Protocol Buffers) или сырые
бинарные данные (до 512 МБ). Если строка содержит число, над ней можно проводить атомарные математические операции.

### Когда применять:

* Классическое кэширование данных: Кэширование профиля пользователя, тяжелых SQL-ответов, конфигураций (SET user:
  123 "{...}" EX 3600).
* Атомарные счетчики и инкременты: Подсчет просмотров страниц, лайков, лимитов в реальном времени (INCR page:views:42).
* Распределенные блокировки (Distributed Locks): Защита от одновременного выполнения операций (SET lock:order:1001 token
  NX PX 10000).
* Rate Limiting: Ограничение количества запросов по API к приложению.

## 2. Hashes (Хэши / Словари)

Представляют собой плоские карты (Maps) вида «поле-значение» внутри одного ключа. Идеально подходят для представления
объектов.

### Когда применять:

* Хранение структурированных объектов: Вместо того чтобы сериализовать пользователя в JSON и класть в String, его
  свойства
  можно разложить по полям хэша (HSET user:123 name "Oleksiy" role "Architect" age 50).
* Оптимизация памяти (Memory Efficiency): Маленькие хэши Redis сжимает на уровне памяти (через zipmap/ziplist). Хранить
  1000 объектов в хэшах гораздо выгоднее по RAM, чем 1000 отдельных Strings.
* Частичное обновление объектов: Когда нужно обновить только одно поле (например, изменить баланс пользователя или время
  последней активности HSET user:123 last_login 1719320000), не перечитывая и не перезаписывая весь объект целиком.

## 3. Lists (Списки)Списки строк, упорядоченные по порядку добавления.

Элементы можно вставлять и удалять как с головы (LPUSH/LPOP), так и с хвоста (RPUSH/RPOP). По сути, это двусвязный
список.

### Когда применять:

* **Простые очереди сообщений (Message Queues)**: Один сервис делает LPUSH queue:jobs task_data, а фоновый воркер
  блокируется командой BRPOP queue:
  jobs 0 и ждет появления задачи. Паттерн Write-Behind часто строится именно на этом.
* **Лента последних событий (Timeline / Activity Feed)**: Хранение последних $N$ действий пользователя (например,
  последние 100 просмотренных
  товаров). Новые добавляются в голову, а всё, что дальше сотого элемента, отсекается командой LTRIM.
* **Архитектурный паттерн Fan-Out / Потоковая обработка**: Перекачка данных между очередями с гарантией доставки с
  помощью BLMOVE (или
  устаревшей RPOPLPUSH).

4. Sets (Множества)
   Неупорядоченная коллекция уникальных строк. Redis позволяет мгновенно проверять вхождение элемента в
   множество (SISMEMBER), а также на лету делать пересечения (SINTER), объединения (SUNION) и разности множеств (SDIFF)
   .

### Когда применять:

* Учет уникальных элементов:Список уникальных IP-адресов, посетивших сайт за день, или уникальных ID пользователей.
* Тегирование (Tagging): Хранение тегов для статей или товаров (SADD article:99:tags "java" "redis" "architecture").
  Можно легко найти статьи, у которых есть пересекающиеся теги.
* Социальные графы / Рекомендательные системы: Списки друзей или подписчиков. Команда SINTER friends:userA friends:userB
  моментально вернет список общих
  друзей двух пользователей.
* Черные/белые списки: Проверка, находится ли токен в списке отозванных (Blacklist) за $O(1)$.

## 5. Sorted Sets (ZSET / Сортированные множества)

Похожи на Sets (все элементы уникальны), но каждому элементу присваивается числовой балл — Score. Элементы автоматически
сортируются по этому баллу. Поиск диапазона элементов выполняется за $O(\log N + M)$.

### Когда применять:

* Лидерборды и рейтинги (Leaderboards): Идеально для онлайн-игр или витрин. Хранит ID игрока и его очки (ZADD game:top
  1500 user:123).
  Позволяет мгновенно выдать топ-10 игроков или узнать место конкретного юзера в общем рейтинге (ZREVRANGE).
* Sliding Window Rate Limiter: Как мы разбирали ранее, где в качестве score используется timestamp в миллисекундах.
* Очереди с приоритетом (Priority Queues): Задачи в очередь кладутся со score, обозначающим важность или точное время
  выполнения (
  Delayed Job Queue). Воркер забирает задачи, у которых score меньше или равен текущему времени.

## 6. Advanced / Специфические типы данных

Эти типы данных используются для узкоспециализированных enterprise-задач.

* HyperLogLog (HLL):
    * Что это: Вероятностная структура данных для подсчета уникальных элементов.
    * Когда применять: Когда нужно посчитать гигантское количество уникальных объектов (например, миллиард поисковых
      запросов за
      сутки) с погрешностью около 1%, но при этом критически важно сэкономить память. Любой HyperLogLog ключ в Redis
      занимает максимум 12 КБ, независимо от того, сколько миллионов элементов в него «закинули».
* Bitmaps (Битовые карты):
    * Что это: Работа со строкой на уровне отдельных битов ($0$ или $1$).
    * Когда применять: Посещаемость и ретеншн (например, «заходил ли пользователь на сайт в 175-й день года»). Один
      бит — один день. Для миллиона пользователей такая
      аналитика занимает копейки в RAM.
* Geospatial (GEO):
    * Что это: Структура, позволяющая хранить координаты (долгота/широта) и искать по ним. Под капотом использует тот же
      Sorted Set.
    * Когда применять: Задачи в стиле такси или доставки: «найти всех курьеров в радиусе 2 километров от этой точки» (
      GEORADIUS / GEOSEARCH).
* Streams (Стримы):
    * Что это: Полноценный append-only лог сообщений. Аналог Apache Kafka, встроенный прямо в Redis.
    * Когда применять: Построение событийно-ориентированной архитектуры (EDA), логирование активности, организация
      сложных очередей с группами потребителей (Consumer Groups), которые могут параллельно вычитывать данные,
      подтверждать обработку (ACK) и масштабироваться.

## Шпаргалка для быстрого выбора:
1. Нужен простой кэш, счетчик или блокировка? $\to$ String
2. Есть объект с кучей полей (например, профиль)? $\to$ Hash
3. Нужна базовая FIFO очередь или история действий? $\to$ List
4. Нужна уникальность, списки друзей, пересечение тегов? $\to$ Set
5. Нужен топ-рейтинг, очередь с приоритетом или скользящее окно? $\to$ Sorted Set
6. Нужна аналитика миллиардов юзеров при минимуме памяти? $\to$ HyperLogLog
7. Делаешь аналог Кафки / шину событий? $\to$ Stream
